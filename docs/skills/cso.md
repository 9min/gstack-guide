# /gstack-cso

> 시크릿 고고학부터 STRIDE 위협 모델링까지, 인프라 전반을 훑는 최고보안책임자(CSO) 모드 감사 스킬.

## 무엇을 하는 스킬인가

cso는 프로젝트의 보안 상태를 인프라 레벨부터 체계적으로 감사합니다. 코드 표면의 취약점만 보는 것이 아니라, 시크릿 관리 이력(secrets archaeology), 의존성 공급망(dependency supply chain), CI/CD 파이프라인 보안, LLM/AI 보안까지 아우르는 깊이 있는 점검을 수행합니다.

감사 프레임워크는 업계 표준을 따릅니다. OWASP Top 10 취약점 항목을 점검하고, STRIDE 위협 모델링을 통해 위협 벡터를 체계적으로 분류합니다. 단순히 알려진 패턴을 찾는 것이 아니라, 실제로 공격이 가능한지 능동적으로 검증합니다.

특히 시크릿 고고학은 git 히스토리 전체를 뒤져서 과거에 커밋되었다가 삭제된 API 키, 비밀번호, 토큰 등을 찾아냅니다. 현재 코드에는 없지만 히스토리에 남아있는 시크릿은 여전히 유출 위험이 있기 때문에 중요한 점검 항목입니다.

## 언제 쓰나

- 프로젝트의 전반적인 보안 상태를 감사하고 싶을 때
- 새로운 의존성을 대량으로 추가한 뒤 공급망 보안을 점검할 때
- CI/CD 파이프라인에 보안 취약점이 없는지 확인할 때
- LLM 연동 기능의 보안(프롬프트 인젝션, 데이터 유출 등)을 점검할 때
- git 히스토리에 노출된 시크릿이 있는지 확인할 때
- 보안 감사 보고서를 작성해야 할 때

## 어떻게 시작하나

`/gstack-cso`를 입력하면 프로젝트 구조를 파악한 뒤 감사를 시작합니다. 시크릿 스캔, 의존성 분석, CI/CD 파이프라인 점검, OWASP/STRIDE 기반 분석 순서로 진행되며, 발견된 이슈는 심각도별로 분류되어 보고됩니다.

## 실제 사용 예시

SaaS 제품의 첫 번째 보안 감사를 진행했습니다. `/gstack-cso`를 실행하니 다섯 가지 카테고리에서 결과가 나왔습니다. 시크릿 고고학에서 6개월 전 커밋에 Stripe API 키가 남아있는 것을 발견했고, 의존성 스캔에서 known vulnerability가 있는 패키지 2개를 찾았습니다. CI/CD 점검에서는 GitHub Actions 워크플로우의 권한이 과도하게 설정된 것이 지적되었고, LLM 보안 점검에서 사용자 입력이 프롬프트에 직접 주입되는 경로가 발견되었습니다. 심각도 순으로 정리된 보고서 덕분에 우선순위를 정해 하나씩 수정할 수 있었습니다.

## 음성 입력

AquaVoice, Whisper 등 음성 입력 도구에서 다음 표현으로 실행할 수 있습니다:

- "run security"
