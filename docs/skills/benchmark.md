# /gstack-benchmark

> PR마다 성능 회귀를 감지하여, 느려진 코드가 배포되기 전에 잡아주는 성능 벤치마크 스킬.

## 무엇을 하는 스킬인가

benchmark는 헤드리스 브라우저 데몬을 활용해 웹 애플리케이션의 성능을 정량적으로 측정합니다. 페이지 로드 시간, Core Web Vitals(LCP, FID, CLS), 리소스 크기 등 핵심 지표에 대한 베이스라인을 설정하고, 이후 변경 사항이 생길 때마다 before/after를 비교합니다.

단순히 한 번 측정하는 것이 아니라, 시간에 따른 성능 추이를 추적합니다. PR 단위로 성능 변화를 감지하기 때문에 어떤 커밋에서 성능이 떨어졌는지 정확히 파악할 수 있습니다.

성능 회귀가 감지되면 구체적인 수치와 함께 어떤 지표가 얼마나 나빠졌는지 알려줍니다. "느려진 것 같다"는 감이 아니라 데이터로 판단할 수 있게 해줍니다.

## 언제 쓰나

- PR을 올리기 전에 성능 영향을 확인하고 싶을 때
- 이미지나 번들 크기가 커진 것 같을 때 정확한 수치를 보고 싶을 때
- Core Web Vitals 점수를 주기적으로 모니터링하고 싶을 때
- 리팩토링 후 성능이 유지되는지 확인할 때
- 새로운 라이브러리를 추가한 뒤 로딩 속도 변화를 측정할 때
- 성능 최적화 작업의 효과를 수치로 증명하고 싶을 때

## 어떻게 시작하나

`/gstack-benchmark`를 입력하면 현재 상태의 성능 베이스라인을 먼저 수집합니다. 이미 베이스라인이 있다면 현재 코드의 성능을 측정한 뒤 이전 베이스라인과 비교하여 변화를 리포트합니다.

## 실제 사용 예시

프로필 페이지에 아바타 업로드 기능을 추가하는 PR을 준비 중입니다. `/gstack-benchmark`를 실행하니 LCP가 1.2초에서 2.8초로 증가했다는 결과가 나왔습니다. 원인을 확인해보니 최적화되지 않은 이미지 라이브러리가 번들 크기를 380KB 늘린 것이었습니다. 이미지를 lazy-load로 변경한 뒤 다시 벤치마크를 돌리니 LCP가 1.3초로 돌아왔습니다. 성능 회귀 없이 PR을 올릴 수 있었습니다.

## 음성 입력

AquaVoice, Whisper 등 음성 입력 도구에서 다음 표현으로 실행할 수 있습니다:

- "speed test"
- "check performance"
