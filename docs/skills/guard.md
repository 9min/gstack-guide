# /gstack-guard

> 위험 명령 경고와 디렉토리 편집 제한을 결합한 최대 안전 모드 스킬.

## 무엇을 하는 스킬인가

guard는 /careful과 /freeze를 하나로 합친 종합 안전 모드입니다. 두 가지 보호 장치를 동시에 활성화합니다. 첫째, `rm -rf`, `DROP TABLE`, `git push --force` 같은 파괴적 명령을 실행하기 전에 경고를 표시합니다. 둘째, 지정된 디렉토리 밖의 파일 편집을 차단합니다.

프로덕션 환경을 다루거나 라이브 시스템을 디버깅할 때, 한 번의 실수가 큰 사고로 이어질 수 있습니다. guard는 이런 상황에서 AI가 실수하더라도 실제 피해가 발생하기 전에 사람이 확인할 수 있는 안전망을 제공합니다.

개별적으로 /careful만 또는 /freeze만 사용할 수도 있지만, 프로덕션 관련 작업에서는 두 가지를 함께 켜는 guard가 권장됩니다. 안전장치는 많을수록 좋습니다.

## 언제 쓰나

- 프로덕션 데이터베이스에 접근해서 디버깅해야 할 때
- 라이브 서버의 설정 파일을 수정해야 할 때
- 배포 스크립트를 디버깅하면서 실수로 force push하면 안 될 때
- 모노레포에서 특정 서비스만 수정하면서 위험 명령도 차단하고 싶을 때
- 민감한 인프라 코드(Terraform, Kubernetes 등)를 다룰 때
- 처음 보는 코드베이스에서 작업할 때 안전하게 시작하고 싶을 때

## 어떻게 시작하나

`/guard src/api`처럼 보호할 디렉토리와 함께 실행합니다. 이후 지정된 디렉토리 밖의 편집이 차단되고, 파괴적 명령 실행 전에 확인 경고가 표시됩니다.

## 실제 사용 예시

프로덕션에서 사용자 데이터 조회가 느려지는 문제를 디버깅하기 위해 `/guard src/api/users`를 실행했습니다. 원인을 추적하던 중 AI가 인덱스를 추가하겠다며 `DROP INDEX` 후 `CREATE INDEX` 쿼리를 제안했습니다. guard가 `DROP INDEX`를 감지하고 경고를 표시했고, 확인해보니 프로덕션 DB에 직접 실행될 수 있는 상황이었습니다. 경고 덕분에 스테이징 환경에서 먼저 테스트한 후 안전하게 적용할 수 있었습니다. 또한 AI가 관련 있어 보이는 `src/api/orders` 파일을 수정하려 했지만 freeze에 의해 차단되어, 문제의 범위를 users 모듈에 집중할 수 있었습니다.
